# W0.10 ŌĆö Background vetter job (cron)

Part of the [pivot plan](README.md). Workstream 0. Moves the existing `discover()` pipeline (LLM rerank + param-shape discovery) out of the hot path and into a scheduled maintenance job.

**Workstream:** 0
**Dependencies:** W0.2, W0.3, W0.9
**Status:** planned

## Acceptance criteria

- A cron job (Cloudflare Cron Triggers or GitHub Actions, choose during ticket execution) runs once per 12h.
- Job runs the existing `discover()` pipeline for each category. New candidates ŌåÆ `probation`. Calls `recomputeScores()` (W0.9). Promotes/demotes per status machine.
- LLM-based param-shape discovery runs **only** here, not in the hot path.
- Manual trigger: `scripts/vet.ts` runs the same job once on demand.
- **Auto-bump `price_usdc` when upstream raises its price.** For every `active` / `probation` row, the vetter does an *unpaid* probe of `resource` (plain `GET`, no `X-PAYMENT` header) and reads `accepts[].maxAmountRequired` from the resulting 402 response. If the real price (`maxAmountRequired` / 1e6) is higher than the stored `price_usdc`, update the row to `max_amount_required * 1.20` (20% buffer to absorb future micro-bumps without re-running the cron). Also propagate the new value back to `data/call_recipes.json` so `seed-registry.ts` doesn't undo the change on the next seed. Rationale: today an upstream price increase causes silent `Payment exceeds maximum allowed value` failures because we still send `maxValueUsd = stored_price`. Observed in real traffic on 2026-06-20 for `orbisapi.com/.../crypto-news-sentiment-api-628b81/news/sentiment` (stored $0.005, real $0.010); the manual fix is now a row in the registry but is fragile to upstream drift.
  - Skip the auto-bump when the new price would exceed a safety ceiling (e.g. `>$0.10`) ŌĆö those go to `probation` for human review instead of being silently accepted.
  - Log the bump (resource, old price, new price) so it shows up in cron output.

## Test spec

- Test: vetter finds a new candidate, inserts as `probation` with `last_vetted_at` set.
- Test: vetter respects `blocked` status ŌĆö doesn't re-attempt.
- Test: vetter probes a resource where stub 402 returns `maxAmountRequired: 20000` (= $0.020) for a row with `price_usdc: 0.005` ŌåÆ row is updated to `0.024` (real ├Ś 1.20), `data/call_recipes.json` is rewritten, bump is logged.
- Test: vetter probes a resource whose stub 402 returns a price above the safety ceiling ŌåÆ row is moved to `probation`, `price_usdc` is **not** auto-bumped, a warning is logged.
- Test: vetter probes a resource where the stored price already covers `maxAmountRequired` ŌåÆ no UPDATE issued, no file write.
