# W0.1 — Postgres infrastructure (Supabase) + initial schema

Part of the [pivot plan](README.md). Workstream 0 (Latency foundation). Shared infra for W0, W1, W2.

**Workstream:** 0
**Dependencies:** none (can run in parallel with W−1)
**Status:** completed — shipped on `feat/w0.1-postgres-neon`. Host = **Neon** (not Supabase): lightest path, pooled endpoint suits Deno Deploy's serverless connection model, plain-Postgres schema = near-zero lock-in. No Docker — local dev uses a Neon dev branch. See [docs/features/postgres-infra.md](../../../features/postgres-infra.md).

## Acceptance criteria

- Supabase project provisioned. Connection string + service-role key in `.env.example`.
- Migrations directory `db/migrations/` with versioned SQL files for tables: `tenants`, `api_keys`, `usage_events`, `service_registry`, `service_observations`, `service_health_durable`.
- Schemas designed to accommodate W1 (tenant FK), W2 (outcome fields, severity contribution), and W4 (forward-compatible outcome labels).
- `src/db/client.ts` exposes a typed Postgres client. Falls back to no-op in test mode.
